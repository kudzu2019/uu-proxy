# UU Proxy

一个功能完整的 Chrome 代理管理扩展：多代理管理、一键切换、代理测速、按域名规则智能分流（全局 / 规则两种模式）、内置规则库、以及**端到端加密的多设备云同步**。后端基于 Cloudflare Worker + D1 + KV，可完全私有部署，数据自主可控。

> 本文档面向零基础用户，Cloudflare 的每一步都有详细说明。

---

## 🚀 两种使用方式（先看这里）

**这个插件的后端（Cloudflare）是完全可选的。** 你可以根据需求二选一：

### 方式一：纯本地离线使用（无需部署，开箱即用）✅ 推荐大多数人

**什么都不用搭建**，下载插件、加载到 Chrome 就能用。以下功能全部可用：

- ✅ 多代理管理、一键切换、代理测速
- ✅ 全局 / 规则两种分流模式
- ✅ 自定义规则（走代理 / 强制直连）、精确域名匹配、通配
- ✅ **配置导入 / 导出**（本地 JSON 备份，换设备靠它搬）
- ✅ 内置规则（首次联网会自动从作者的公共接口拉取一份；也可跳过）

**唯一区别**：没有"账号登录 / 云端自动同步 / 在界面里管理云端内置规则"。多设备之间用**导出/导入 JSON** 手动同步即可，完全够用。

> 如果你只是想在自己电脑上管理代理、做分流，**直接看 [第二部分：安装浏览器插件](#第二部分安装浏览器插件)，跳过整个第一部分。**

### 方式二：私有部署后端（进阶，需要云同步的人）

自己在 Cloudflare（免费）部署一个后端，额外获得：

- ☁️ 账号注册登录、**端到端加密的多设备云同步**（改动自动上传，另一台设备登录即拉回）
- ☁️ 在插件界面里**增删云端内置规则**并同步到所有设备

数据存在**你自己的** Cloudflare 账号里，自主可控。适合有多台设备、想自动同步、或想长期维护自己内置规则库的人。

> 想要云同步 → 看 [第一部分：私有部署后端](#第一部分私有部署后端cloudflare)。

**简单说：想要多设备自动同步就部署后端；否则纯本地用 + 导入导出就够了。**

## 功能特性

- **多代理管理**：添加 / 编辑 / 删除代理，支持 `http` / `https` / `socks5` 三种协议，支持用户名密码认证。
- **一键解析**：粘贴 `ip:端口:账号:密码`、`账号:密码@ip:端口` 等多种格式自动填表。
- **代理测速**：一键测试节点延迟。
- **两种分流模式**
    - **全局模式**：所有网站走当前代理（白名单除外）。
    - **规则模式**：只有命中规则的域名走代理，其余直连。
- **精确域名匹配**：后缀匹配（`google.com` 只命中它及其子域，不会误伤 `notgoogle.com`），支持 `*.google` 通配；同层内**更具体（更长）的规则优先**。
- **内置规则库**：AI（OpenAI/Claude/Grok 等）、Google 全家桶（含 195 个国别域名）、社交、流媒体、开发工具等，云端可管理、可同步。
- **国内域名直连**：在系统规则里添加 `*.cn`、`*.中国` 等直连规则即可（中文域名会自动转 punycode）。
- **自定义规则**：自己添加"走代理"或"强制直连"的域名，优先级高于内置规则。
- **系统规则管理**：输入管理密钥后可在界面里增删云端内置规则，所有设备同步。
- **端到端加密云同步**：账号密码在本地派生密钥，配置加密后上传，**服务器看不到你的代理密码**；支持任意多设备。
- **导入 / 导出**：配置备份为本地 JSON。

## 项目架构

```
┌──────────────────────────┐        HTTPS         ┌─────────────────────────────┐
│   Chrome 扩展（前端）     │  ◄───────────────►   │   Cloudflare Worker（后端） │
│                          │                      │                             │
│  · popup.*  界面         │   注册/登录/同步     │   · D1 数据库  → 存账号      │
│  · background.js  PAC引擎 │   内置规则读写       │   · KV 存储    → 存加密配置  │
│  · data.js  静态域名表   │                      │                    + 内置规则│
└──────────────────────────┘                      └─────────────────────────────┘
```

- **前端**：纯 HTML/CSS/JS 的 Chrome MV3 扩展。核心是 `background.js` 里的 **PAC 脚本生成引擎**，根据你的规则和模式动态生成分流脚本交给 Chrome。
- **后端**：一个 Cloudflare Worker，提供账号注册登录、加密配置同步、内置规则读写等接口。数据存在你自己的 D1（账号）和 KV（配置 + 内置规则）里。
- **后端是可选的**：不部署后端，插件的代理管理、分流、测速、导入导出都能用；只有"登录 / 云同步 / 系统规则云端管理"需要后端。

# 第一部分：私有部署后端（Cloudflare）

> ⚠️ **这一部分是可选的！** 只有需要「账号登录 / 云端自动同步 / 界面内管理云端内置规则」时才需要。
> 如果你只想本地管理代理和分流，请**直接跳到 [第二部分](#第二部分安装浏览器插件)**，本插件无需任何后端即可正常使用。

> 这一部分教你把后端部署到**你自己的** Cloudflare 账号。全程免费额度足够个人使用。预计 15–20 分钟。

## 0. 准备工作

你需要：

- 一个能收邮件的邮箱（注册 Cloudflare 用）。
- 一台能上网的电脑，安装了 Chrome 浏览器。
- （可选）一个你自己的域名 —— 如果你想用 `proxy.你的域名.com` 这种自定义地址而不是 Cloudflare 送的 `xxx.workers.dev`。没有也完全可以，用送的免费域名即可。

**几个名词先了解一下**（不懂也没关系，照做就行）：

| 名词                | 通俗解释                                                                                                                                      |
| ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| **Worker**          | 一段跑在 Cloudflare 服务器上的代码，就是我们的"后端程序"。                                                                                    |
| **D1**              | Cloudflare 提供的 SQL 数据库，我们用它存**账号**。                                                                                            |
| **KV**              | Cloudflare 提供的键值存储，我们用它存**加密配置**和**内置规则**。                                                                             |
| **绑定（Binding）** | 把 D1 / KV "接"到 Worker 上，让 Worker 代码能访问它们。绑定时要起一个**变量名**，我们代码里固定用 `DB` 和 `CONFIG_KV`，**名字必须一模一样**。 |
| **密钥（Secret）**  | 保存在 Cloudflare 里的机密字符串，代码能读到但别人看不到。我们要设两个。                                                                      |

---

## 1. 注册 Cloudflare 账号

1. 打开 https://dash.cloudflare.com/sign-up
2. 输入邮箱和密码，注册。
3. 去邮箱点验证链接，激活账号。
4. 登录后进入 Cloudflare 控制台（Dashboard）。

> 如果你有自己的域名想用（第 8 步），可以在控制台左侧「Websites / 网站」里先把域名添加进来（Add a site），按提示把域名的 NS 服务器改到 Cloudflare。**没有域名可跳过。**

---

## 2. 创建 Worker

1. 在 Cloudflare 控制台左侧菜单，点 **Workers & Pages（Workers 和 Pages）**。
2. 点右上角 **Create（创建）** → 选 **Create Worker（创建 Worker）**。
3. 给 Worker 起个名字，比如 `proxy-soft`（这个名字会成为你的地址 `proxy-soft.你的账号.workers.dev` 的一部分，只能用小写字母、数字、连字符）。
4. 点 **Deploy（部署）**。此时会部署一个默认的示例代码，先不管它，我们后面替换。
5. 部署成功后，记下你的 Worker 地址，形如：
    ```
    https://proxy-soft.你的账号.workers.dev
    ```
    点页面上的地址能打开（显示示例内容）就说明 Worker 建好了。

---

## 3. 创建 D1 数据库并建表

**3.1 创建数据库**

1. 控制台左侧菜单 → **Storage & Databases（存储和数据库）** → **D1 SQL Database**。
   （不同版本界面里也可能在 Workers & Pages 页面的 D1 标签下。）
2. 点 **Create（创建）**。
3. 数据库名字填 `uu-proxy-db`（随意，记住即可），点创建。

**3.2 建表**

1. 进入刚创建的数据库，找到 **Console（控制台 / 查询）** 标签页。
2. 把下面这条 SQL **单独**粘进去，点执行（Execute / Run）：

    ```sql
    CREATE TABLE IF NOT EXISTS users (username TEXT PRIMARY KEY, auth_hash TEXT NOT NULL, server_salt TEXT NOT NULL, token TEXT, created_at INTEGER NOT NULL, updated_at INTEGER NOT NULL);
    ```

3. 执行成功后，**再单独**粘这条，点执行：

    ```sql
    CREATE INDEX IF NOT EXISTS idx_users_token ON users(token);
    ```

    > ⚠️ 一定要**一条一条**执行，不要两条一起粘、也不要带 `--` 注释，否则 D1 控制台可能报 `incomplete input` 错误。

建好后，在数据库的 Tables（表）里能看到 `users` 表就对了。

---

## 4. 创建 KV 命名空间

1. 控制台左侧 → **Storage & Databases（存储和数据库）** → **KV**。
2. 点 **Create a namespace（创建命名空间）**。
3. 名字填 `uu-proxy-kv`（随意），创建。

创建好即可，里面暂时是空的，后面代码会往里写数据。

---

## 5. 给 Worker 绑定 D1 和 KV

> 这一步是**最容易出错**的地方，请务必让变量名和下面完全一致。

1. 回到 **Workers & Pages** → 点开你的 Worker（`proxy-soft`）。
2. 点 **Settings（设置）** 标签 → 找到 **Bindings（绑定）** 区域。

**5.1 绑定 D1**

3. 在 Bindings 里点 **Add（添加）** → 选 **D1 database**。
4. **Variable name（变量名）** 填：`DB` ← **必须是大写的 DB，不能改**
5. **D1 database** 选你第 3 步创建的 `uu-proxy-db`。
6. 保存。

**5.2 绑定 KV**

7. 再点 **Add** → 选 **KV namespace**。
8. **Variable name（变量名）** 填：`CONFIG_KV` ← **必须一模一样**
9. **KV namespace** 选你第 4 步创建的 `uu-proxy-kv`。
10. 保存。

> 变量名错了（比如写成小写 `db`、或 `CONFIG-KV`）会导致 Worker 读不到数据库，注册登录全部失败。请再三核对。

---

## 6. 设置两个密钥

在同一个 **Settings** 页面，找到 **Variables and Secrets（变量和密钥）** 区域，添加两个 **Secret（加密类型，不是 Plaintext）**：

**6.1 `TOKEN_SECRET`**

- 名称：`TOKEN_SECRET`
- 值：一段随机长字符串。生成方法（任选）：
    - 在电脑终端跑：`openssl rand -hex 32`
    - 或在线随机字符串生成器，弄一段 64 位十六进制。
- **作用**：给用户登录后的"登录凭证（token）"做数字签名。它保证 token 不能被伪造，并让**同一账号在任意多台设备同时登录、互不掉线**。
- **注意**：一旦设定，**不要随便更换**——更换后所有已登录用户的 token 会立即失效，需要重新登录。

**6.2 `ADMIN_KEY`**

- 名称：`ADMIN_KEY`
- 值：另一段随机长字符串（同样 `openssl rand -hex 32` 生成，**不要和 TOKEN_SECRET 相同**）。
- **作用**：管理**内置规则库**（系统规则）的密码。谁拿着它，就能往云端内置规则里加 / 删域名。插件里的"管理"功能就是用它解锁的。
- **注意**：这是管理员口令，**自己保管好，不要泄露、不要提交到 GitHub**。

> ⚠️ 添加 / 修改 Secret 后，**必须重新部署一次 Worker**（见下一步），密钥才会注入到运行中的程序。

---

## 7. 粘贴 Worker 代码并部署

1. 回到 Worker 页面，点 **Edit code（编辑代码）**（或右上角 `</> Quick edit`）。
2. 把编辑器里默认的代码**全部删除**。
3. 打开本仓库的 **`worker.js`** 文件，把里面**全部内容**复制，粘贴进编辑器。
4. 点右上角 **Deploy（部署）**。

部署成功后，后端代码就上线了。

> `worker.js` 里已经写好了所有接口：注册、登录、配置上传下载、内置规则读写。它要求绑定名为 `DB`、`CONFIG_KV`，密钥为 `TOKEN_SECRET`、`ADMIN_KEY`——也就是你前面几步设置的那些。

---

## 8. 可选：绑定自定义域名

如果你想用 `proxy.你的域名.com` 而不是 `xxx.workers.dev`：

**前提**：该域名已经在你这个 Cloudflare 账号里（第 1 步末尾提到的 Add a site）。

1. Worker 页面 → **Settings** → **Domains & Routes（域名和路由）** 或 **Triggers（触发器）** → **Custom Domains（自定义域）**。
2. 点 **Add Custom Domain**，输入你想用的完整域名，例如 `proxy.你的域名.com`。
3. Cloudflare 会自动为它签发证书、配置解析，等状态变成 Active 即可。

之后这个域名就等价于你的 Worker 地址。

> **为什么用自定义域更好**：`workers.dev` 域名在国内有时不稳定；自定义域（尤其套 Cloudflare CDN）通常可达性更好。

---

## 9. 灌入内置规则

内置规则（AI/Google/流媒体等域名）要先写进 KV，插件才能拉到。用管理密钥执行一次即可。

1. 确认你已经设好 `ADMIN_KEY`（第 6 步）。
2. 下载本仓库的 **`builtin.json`**，放到电脑某个文件夹，在该文件夹打开终端。
3. 执行（把 `<你的ADMIN_KEY>` 换成第 6 步那个，把地址换成你的 Worker 地址或自定义域）：

    **Windows PowerShell：**

    ```powershell
    curl.exe -X PUT https://你的地址/builtin -H "Content-Type: application/json" -H "X-Admin-Key: <你的ADMIN_KEY>" --data "@builtin.json"
    ```

    **Mac / Linux：**

    ```bash
    curl -X PUT https://你的地址/builtin -H "Content-Type: application/json" -H "X-Admin-Key: <你的ADMIN_KEY>" --data @builtin.json
    ```

4. 返回 `{"ok":true,"version":...}` 即成功。

> 以后想加内置域名，不用再改这个文件——直接在插件里用"管理"功能加即可（见使用教程）。

---

## 10. 验证后端

在浏览器直接打开（GET 是公开的）：

```
https://你的地址/builtin
```

能看到一大串 JSON（包含 `proxy` 和 `direct` 两个数组）就说明**数据库、KV、代码、内置规则全部就绪**。

也可以用命令行完整自测（把地址和 KEY 换成你的）：

```bash
# 注册一个测试账号（authHash 这里随便填个十六进制串模拟）
curl -X POST https://你的地址/register -H "Content-Type: application/json" -d "{\"username\":\"test01\",\"authHash\":\"abc123\"}"
# 登录，应返回 token
curl -X POST https://你的地址/login -H "Content-Type: application/json" -d "{\"username\":\"test01\",\"authHash\":\"abc123\"}"
```

返回里有 `"ok":true` 和 `token` 就说明后端完全正常。

# 第二部分：安装浏览器插件

## 1. （仅私有部署者需要）修改插件里的后端地址

> **纯本地使用可跳过此步**，直接看下面的第 2 步加载插件即可（登录/同步功能不用就不受影响）。

如果你部署了自己的后端（第一部分），插件默认连接的是作者的后端地址，**必须改成你自己的**：

1. 用文本编辑器打开 `background.js`，把第 5 行：
    ```js
    const WORKER = "https://proxy-soft.19920806.xyz";
    ```
    改成你自己的 Worker 地址（第 2 步的 `workers.dev` 或第 8 步的自定义域），**结尾不要带斜杠**：
    ```js
    const WORKER = "https://你的地址";
    ```
2. 同样地，打开 `popup.js`，把第 5 行的 `const WORKER = ...` 改成一样的地址。

> 两个文件都要改，且地址必须一致。

## 2. 加载扩展到 Chrome

1. 打开 Chrome，地址栏输入 `chrome://extensions` 回车。
2. 打开右上角 **开发者模式（Developer mode）**。
3. 点 **加载已解压的扩展程序（Load unpacked）**。
4. 选择整个 `uu-proxy` 文件夹。
5. 加载成功后，把插件图标固定到工具栏。

> 以后修改了插件代码（比如换地址），要回到这个页面点该扩展的**刷新/重新加载**按钮；改动较大时建议**移除后重新加载**。

# 第三部分：使用教程

## 添加代理

1. 点插件图标 → **代理列表** 标签 → **添加代理**。
2. 可以手动填协议 / IP / 端口 / 账号密码，也可以在"一键解析"里粘贴常见格式自动填表。
3. 保存后点卡片选中它（显示"当前使用"），打开底部右侧的**总开关**即生效。
4. 点卡片上的 **测速** 可测该节点延迟。

> **注意**：Chrome 不支持带账号密码的 **socks5** 代理认证（浏览器限制）。带认证请用 `http` / `https` 协议；socks5 请用免认证节点。

## 全局模式 / 规则模式

底部有两个单选：

- **全局模式**：除白名单外，所有流量走代理。
- **规则模式**：只有命中规则的域名走代理，其余直连（适合国内正常上网、只有特定网站走代理）。

## 自定义规则（自定义规则 标签）

- 顶部选 **代理** 或 **直连**，输入域名（支持空格 / 逗号 / 分号 / `|` / `#` 批量），点添加。
    - 类型选**代理** = 该域名走代理；选**直连** = 强制直连（即"白名单"）。
- 必须填**完整域名**（如 `google.com`），不支持半截关键词。
- 中间的下拉可按"全部 / 已启用 / 已禁用"过滤，旁边搜索框可搜规则。
- 每条可**启用/禁用/删除**；**点域名文字可复制**。
- 自定义规则**优先级高于内置规则**，可用来覆盖内置（比如把某个默认走代理的域名改成直连）。

## 系统规则与管理模式（系统规则 标签）

- 这里是**内置规则库**，默认只读，可按状态筛选、可本机启用/禁用某条。
- 想**增删云端内置规则**（所有设备同步）需要管理权限：
    1. 先在设置里**登录账号**。
    2. 账号那一行点 **管理** → 输入你的 `ADMIN_KEY` → 解锁（本机记住，下次自动解锁）。
    3. 解锁后系统规则页顶部出现添加框，可加/删内置域名，直接写入云端。
    4. 点「同步内置规则」可从云端拉取最新内置库。

## 账号与云同步（设置 标签）

- **未登录**：显示"登录 / 注册"。
- **注册/登录**：账号 3–32 位；密码用于**端到端加密**——它在本地派生出加密密钥，配置加密后才上传，**服务器看不到你的代理密码**。
    - ⚠️ **忘记密码 = 云端配置无法恢复**（这是端到端加密的代价）。
- **自动同步**：登录后，任何改动（增删代理、切换、改规则、改设置）会在约 2 秒后自动加密上传。
- **手动同步**：`⬆ 上传到云端` / `⬇ 从云端恢复`，覆盖前会提示确认。
- **多设备**：另一台设备登录同一账号即可拉回配置；同一账号可任意多设备同时在线。

## 导入 / 导出（设置 标签）

- **导出 JSON**：把当前配置存成本地文件（**含代理账号密码明文，请妥善保管**）。
- **导入 JSON**：从文件恢复。
- 不依赖登录，是纯本地备份手段。

## 工作原理

### 分流是怎么实现的

`background.js` 根据你的规则和模式，**动态生成一段 PAC 脚本**（Proxy Auto-Config）交给 Chrome。浏览器每访问一个域名，就用这段脚本判断"走代理还是直连"。

### 匹配优先级

**全局模式**（命中即停）：

1. 内网 IP / 本机 / localhost → 直连
2. 自定义规则（直连/代理，同层最长匹配优先）→ 命中即按其类型
3. 内置白名单（直连）→ 直连
4. 其余 → 走代理

**规则模式**（命中即停）：

1. 内网 IP / 本机 / localhost → 直连
2. 自定义规则（直连 / 代理）→ 命中即按其类型；同一域名下**更具体（更长）的规则优先**
3. 内置规则（直连 / 代理，含 Google 国别域名）→ 命中即按其类型；同层最长匹配优先
4. 都不命中 → 直连

> 因此：**"自定义"整体优先于"内置"**；同一层内**更具体（更长）的规则优先**——例如内置里 `bing.com` 走代理、`cn.bing.com` 走直连，访问 `cn.bing.com` 会命中更长的那条走直连。想让某个国内域名/IP 强制走代理，把它加进**自定义代理**即可（自定义优先于内置的 `*.cn` 等直连规则）。国内直连不再是独立开关，改为在系统规则里添加 `*.cn`、`*.中国` 等直连规则实现。

### 匹配算法

- **后缀匹配**：规则 `google.com` 命中 `google.com` 与 `x.google.com`，不命中 `notgoogle.com`。
- **通配**：`*.google` 命中整个 `.google` 顶级域；国内直连用 `*.cn`、`*.中国` 这类规则实现。
- **最长匹配优先**：同一层里若父域和子域规则冲突（如 `cc.com` 代理、`xx.cc.com` 直连），以更具体（更长）的那条为准。
- 中文域名规则会自动转 **punycode**（PAC 只接受 ASCII）。

### 加密同步

登录密码在浏览器本地用 PBKDF2 派生两把钥匙：一把**登录哈希**（上传给服务器验证身份），一把**加密密钥**（只留本地，用 AES-GCM 加密配置）。服务器只存密文，无法解密你的代理密码。

## 常见问题与排错（FAQ）

**Q：注册/登录报 `HTTP 404` 或 `not found`？**
Worker 代码没部署对。确认 `worker.js` 全部内容已粘贴并 Deploy。

**Q：报错提到 `DB` / `CONFIG_KV` / server error？**
绑定名不对。回第 5 步核对：D1 绑定名必须是 `DB`，KV 绑定名必须是 `CONFIG_KV`。

**Q：`env.ADMIN_KEY` 或 `TOKEN_SECRET` 读到是空的？**
改了密钥没重新部署。设置密钥后**必须再 Deploy 一次**。

**Q：`Failed to fetch`？**
后端地址填错、或该地址此刻不可达、或正开着有问题的代理。先直连、并在浏览器直接打开 `https://你的地址/builtin` 看是否返回 JSON。

**Q：设置了代理但任何网站都打不开？**
若你用的是**明文 HTTP 代理**穿墙，域名会暴露被墙重置（`ERR_CONNECTION_RESET`）。请改用加密隧道客户端在本地暴露的 `127.0.0.1` 端口，或给代理套 TLS 变成 https 代理。

**Q：https 代理连上但网页全部报 `ERR_HTTP2_PROTOCOL_ERROR`？**
你的 https 代理在 TLS 里协商了 HTTP/2，而普通 http 代理不支持 h2 CONNECT。解决：在服务端（如 Xray）的 TLS 设置里强制 `"alpn": ["http/1.1"]`。

**Q：后台报 `pacScript.data supports only ASCII`？**
规则里有非 ASCII 字符。本插件已自动把中文域名转 punycode；若仍出现，请更新到最新代码。

**Q：改了规则不生效？**
Chrome 缓存了旧连接。到 `chrome://net-internals/#sockets` 点 Flush socket pools，或重启 Chrome。

**Q：socks5 填了账号密码连不上？**
Chrome 不支持 socks5 代理认证，请改用 http/https，或用免认证 socks5。

## 安全说明

- **私有部署后**务必修改插件里的 `WORKER` 地址为你自己的后端，否则会连到他人服务器。
- `ADMIN_KEY`、`TOKEN_SECRET`、Cloudflare API Token **绝不要提交到 GitHub**。发布前请检查代码与提交历史。
- 导出的 JSON 含代理密码明文，注意保管。
- 端到端加密只保护"传输和服务器存储"，本机上代理密码是明文（扩展要用它连代理，无法避免）。
- 忘记账号密码将无法恢复云端配置。
